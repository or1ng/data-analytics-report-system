🏷️
事件命名规范
统一的命名格式是数据一致性的第一道防线，杜绝"同一事件多个名字"的问题
📋 命名格式
{模块}_{对象}_{动作}
示例：order_create_success、user_login_fail、cart_item_remove、page_view_duration

规则：全部小写，单词间用下划线分隔，禁止用驼峰/中文/缩写（URL除外）。
规范项要求正确示例错误示例
模块前缀按业务域划分，统一在注册表中维护order_、user_、payment_、content_ord_、usr_、pay_（缩写不清）
对象名名词，代表被操作的业务实体order、coupon、article、videoo、cp、art（不可读）
动作名动词/状态，代表发生了什么create、click、submit、fail、successadd（语义模糊，add to cart? add user?）
属性命名与事件名同级规范，小写下划线item_id、source_channel、is_vipitemId、sourceChannel（驼峰）
版本管理事件结构变更时递增版本号，保留旧版本兼容期order_create_v2直接修改order_create的字段
🔍 命名冲突检查清单
新增事件上线前必须检查：① 埋点注册表中是否已存在语义相同的事件名；② 是否与已有事件的属性名冲突；③ 是否符合命名格式规范。检查通过后在注册表中登记，未登记的事件不得上线。

埋点设计原则
6条核心原则，确保埋点数据可追溯、可分析、可信赖

📌 核心原则

1
可追溯性 — 每条埋点日志必须包含：事件时间戳（毫秒级UTC）、用户唯一标识（device_id + user_id）、会话ID（session_id）、页面来源（referrer）
2
唯一性 — 同一用户行为不重复记录。通过事件ID（event_id = hash(user_id + event_name + timestamp + random)）实现幂等去重
3
时间戳标准化 — 统一使用UTC+0毫秒级Unix时间戳，客户端记录发生时间，服务端记录接收时间，两者之差即为传输延迟
4
属性结构化 — 事件属性必须是结构化字段（String/Number/Boolean/Array），禁止将业务数据序列化为JSON字符串塞进单个字段
5
版本管理 — 事件结构变更时必须递增版本号，旧版本保留至少30天兼容期。禁止直接修改已上线事件的字段定义
6
去重机制 — 数据管道层面通过 event_id 做精确去重；分析层面按 user_id + event_name + time_window 做逻辑去重（如同一用户1秒内多次点击只计1次）
🔍
埋点质量监控
上线前必检 + 上线后持续监控，防止埋点问题污染下游报告
🚀 上线前必检项

覆盖率检查

核心页面/按钮是否100%覆盖埋点？对照埋点设计文档逐页验证。

格式校验

事件名是否符合命名规范？属性字段类型是否与注册表一致？必填字段是否有值？

字段完整性

所有预定义属性字段是否存在？枚举值是否在允许范围内？

客户端兼容性

iOS/Android/Web三端埋点行为是否一致？弱网/离线场景是否正确处理？

📊 上线后监控指标

日级埋点日志量波动

各核心事件的日日志量环比波动不超过±15%，超过阈值触发告警（排除业务因素后）。

异常值检测

检测属性字段的异常分布：数值类字段出现负数/极端值、枚举类字段出现未知值、必填字段空值率突增。

客户端版本分布

监控各App版本上报的埋点数据占比，旧版本占比过高需推动升级。

数据延迟监控

埋点从客户端发生到数据仓库可查询的端到端延迟，日报类数据≤6小时，实时类≤5分钟。

💡 建立埋点质量看板
建议搭建一个独立的"埋点质量监控看板"，实时展示各核心事件的日志量趋势、异常率、延迟分布。当检测到异常时，自动在数据团队群发送告警，标注影响范围（哪些下游报告/指标可能受影响），并在埋点注册表中标记该事件状态为"异常"。

🛡️ 需求评审与范围防护

什么需求接：决策导向、数据可获取、有明确交付时间的标准化需求。

什么需求不接："随便看看""把所有数据都分析一下"——退回去要求明确范围。

什么需求需裁剪：业务方要10个指标但时间只够做5个——协商优先级，明确告知"这期先做Top 5，其余下期补"。

范围蔓延应对：需求确认后如业务方新增指标/维度，评估是否影响交付时间。影响>20%工时的，单独开一期处理，不在当前报告中追加。

📊 高管月报 · 结构化模板
📋 使用说明
每个模块标注了「必填/选填」和填写规则。填写时先写结论（1-2句话），再用数据支撑。篇幅控制在2页A4以内。
1
报告头部【必填】

报告期次、生成时间、数据截止日期、编制人

2
一句话总结【必填】

用一句话概括本月最重要的业务结论。格式："本月[指标]达到[数值]，[同比/环比][变化]，主要因为[原因]。"

3
核心指标看板【必填】
  • 3-5个北极星指标，包含：当月值、目标值、达成率、环比变化、同比变化
  • 用红黄绿灯标记健康状态（绿≥90%、黄80-90%、红＜80%）
  • 与上期相比有显著变化的指标，必须标注原因
4
重点发现与归因【必填】
  • 最多列出3个重点发现，每个包含：发生了什么→为什么→影响多大→建议做什么
  • 区分"好消息"和"需关注"两类
  • 每个发现必须附量化数据（不说"大幅提升"，说"提升23%"）
5
风险与机会【必填】

📝 示例：研究问题与方法

📝 示例：因果效应与结论
首单转化率：政策使新用户首单转化率即时提升3.2pp（从8.7%到11.9%），统计显著（p<0.001，95%CI[2.1pp, 4.3pp]）。效应在政策后30天内稳定，未出现衰减。
7日留存率：提升1.8pp（从42%到43.8%），边际显著（p=0.048）。
成本效益：优惠券面额¥20/人，按新用户日均1.2万人计算，月成本¥720万。带来的额外首单GMV约¥980万/月（客单价¥135×额外转化人数），扣除优惠券成本后净收益约¥260万/月。
建议：继续执行该政策。建议测试将优惠券面额从¥20降至¥15（A/B），评估是否能保持相近效应的同时降低成本。

倾向得分匹配		观察组数据，通过匹配创造"伪对照组"选择偏差可被观测变量解释灵活，可处理多个混淆变量只能控制观测到的混淆

-- N日留存率（以次日留存为例） WITH cohort AS ( SELECT user_id, MIN(dt) AS first_day FROM user_active_log GROUP BY user_id ) SELECT c.first_day AS cohort_date, COUNT(DISTINCT c.user_id) AS new_users, COUNT(DISTINCT a.user_id) AS retained, ROUND(COUNT(DISTINCT a.user_id) / COUNT(DISTINCT c.user_id) * 100, 1) AS retention_rate FROM cohort c LEFT JOIN user_active_log a ON c.user_id = a.user_id AND a.dt = DATE_ADD(c.first_day, 1) WHERE c.first_day BETWEEN '${start}' AND '${end}' GROUP BY c.first_day ORDER BY c.first_day;
-- RFM用户分群（最近购买R、频率F、金额M） WITH rfm_raw AS ( SELECT user_id, DATEDIFF('${date}', MAX(order_date)) AS recency, COUNT(DISTINCT order_id) AS frequency, SUM(payment_amount) AS monetary FROM orders WHERE order_date >= DATE_SUB('${date}', 90) GROUP BY user_id ), rfm_score AS ( SELECT *, NTILE(5) OVER(ORDER BY recency ASC) AS r_score, NTILE(5) OVER(ORDER BY frequency DESC) AS f_score, NTILE(5) OVER(ORDER BY monetary DESC) AS m_score FROM rfm_raw ) SELECT user_id, recency, frequency, monetary, r_score, f_score, m_score, CASE WHEN r_score>=4 AND f_score>=4 AND m_score>=4 THEN '高价值' WHEN r_score>=3 AND f_score<=2 THEN '流失风险' WHEN r_score<=2 AND f_score>=3 THEN '召回候选' ELSE '一般用户' END AS segment FROM rfm_score;
-- 转化漏斗分析 WITH funnel AS ( SELECT user_id, MAX(CASE WHEN event = 'view' THEN 1 ELSE 0 END) AS step_view, MAX(CASE WHEN event = 'add_to_cart' THEN 1 ELSE 0 END) AS step_cart, MAX(CASE WHEN event = 'pay' THEN 1 ELSE 0 END) AS step_pay FROM event_log WHERE dt = '${date}' GROUP BY user_id ) SELECT SUM(step_view) AS view_users, SUM(step_cart) AS cart_users, SUM(step_pay) AS pay_users, ROUND(SUM(step_cart)/SUM(step_view)*100,1) AS view_to_cart, ROUND(SUM(step_pay)/SUM(step_cart)*100,1) AS cart_to_pay, ROUND(SUM(step_pay)/SUM(step_view)*100,1) AS view_to_pay FROM funnel;

import numpy as np from scipy import stats def ab_test_analysis(control, treatment, alpha=0.05): """A/B测试结果分析（比例类指标）""" n_c, conv_c = len(control), control.sum() n_t, conv_t = len(treatment), treatment.sum() p_c, p_t = conv_c/n_c, conv_t/n_t # Z检验 p_pool = (conv_c+conv_t) / (n_c+n_t) se = np.sqrt(p_pool*(1-p_pool)*(1/n_c+1/n_t)) z_stat = (p_t - p_c) / se p_value = 2 * (1 - stats.norm.cdf(abs(z_stat))) ci = (p_t-p_c) + np.array([-1,1]) * 1.96 * se return { 'control_rate': p_c, 'treatment_rate': p_t, 'abs_lift': p_t-p_c, 'rel_lift': (p_t-p_c)/p_c, 'z_stat': z_stat, 'p_value': p_value, 'ci_95': ci, 'significant': p_value < alpha }

🔐 权限与版本控制

同一份报告，不同角色看到不同版本：

① CEO/VP版（摘要版）：1页A4，3句话总结 + 核心指标 + 需决策事项。重点在于"该做什么"。

② 总监版（完整版）：完整报告，含分析过程、图表、建议。重点在于"为什么这样做"。

③ 分析师版（底层数据版）：完整报告 + 查询SQL + 原始数据路径 + 方法论附录。重点在于"可复现、可追溯"。

实现方式：在BI平台或飞书文档中设置权限分层，按角色标签自动分发对应版本。

🤖 推送自动化配置

定时任务配置规范：

① 调度工具：使用Airflow/Prefect配置DAG，每个报告对应一个DAG任务。

② 上游依赖：报告推送DAG依赖数据管道ETL完成状态，ETL未完成则自动等待并标注延迟。

③ 失败重试：推送失败自动重试3次（间隔5分钟），3次仍失败则升级告警至数据工程负责人。

④ 推送验证：推送后自动检查报告是否生成成功（文件大小/页数是否正常），异常则不推送并发告警。

推送失败通知：在数据团队群自动发送告警，包含失败原因、影响报告、预计恢复时间。

💬 反馈收集机制
每份推送的报告底部嵌入一个"有反馈"入口（飞书快捷回复 / 邮件中的反馈链接）。收集两类反馈：① 报告有用吗？（有用/一般/没用，1秒完成）；② 建议改进什么？（自由文本，选填）。反馈数据自动汇总到「报告效果度量」模块，每季度分析一次，持续优化报告内容。

🛡️
误读与误用防护机制
数据分析报告最常见的灾难不是数据错了，而是读者理解错了

📊 数值呈现规范

✓
所有同比/环比变化必须标注基准值（不说"增长20%"，说"从100增至120，增长20%"）
✓
图表Y轴截断时必须标注（如Y轴不从0开始，需在图表旁注明）
✓
百分比变化与百分点变化严格区分（"提升5%" vs "提升5个百分点"）
!
小样本数据（n＜30）需标注"样本量较小，结论仅供参考"

⚠️ 因果结论规范

✓
未经因果推断方法验证的关联性结论，必须用"相关"而非"导致"
✓
"显著变化"需附判定标准（如"偏离历史均值超过2个标准差"）
✓
预测类结论必须标注置信区间或预测不确定性
!
避免绝对化表述（"一定会增长"→"预计将增长，但存在不确定性"）

📝 报告使用约束

✓
报告头部标注"内部使用"或"可对外分享"，避免误传
✓
报告中的敏感数据（如用户数、营收）标注密级
✓
时效性数据标注有效期（如"数据截至2026年3月31日"）
!
截屏分享时，确保包含完整上下文（含日期、口径说明）
🚨 误读典型案例
错误表述："促销活动导致GMV增长了30%"
问题："导致"暗示因果关系，但实际上可能只是因为恰逢国庆假期消费旺季。
正确表述："促销活动期间GMV同比增长30%，较促销前基线提升15%。但需注意同期为国庆消费旺季，历史同期自然增幅约12%。基于ITS分析控制季节性后，促销的净效应估计约为3-5%（p=0.048）。"

数据隐私与合规规范
确保报告中的数据使用符合《个人信息保护法》、GDPR等法规要求
合规要求具体规范实施方式
数据最小化报告中只展示分析所需的最少数据，不展示可识别个人身份的原始数据用户分析报告中使用用户ID而非手机号/姓名
脱敏处理涉及个人数据时，必须脱敏或聚合（≥N人时才能展示分组数据）K值阈值：每组≥5人方可展示；5人以下标注"数据不足"
敏感标记涉及营收、用户数、战略数据等商业敏感信息的报告标注密级密级：公开 / 内部 / 机密 / 绝密，对应不同的分发范围
数据留存报告底稿中涉及的个人信息，按法规要求设定留存期限用户行为数据：原始日志保留6个月，聚合数据可永久保留
对外分享含内部数据的报告对外分享前，必须经过合规审查提交法务/合规团队审核，获取"可对外分享"授权
第三方数据使用第三方数据源（如广告后台、竞品工具）需确认授权范围在报告中标注数据来源和授权范围，避免超出授权使用
