# 数据分析报告体系设计方案 · v1.1 版本更迭说明

> **文档版本**: v1.1
> **基准版本**: v1.0（盲区补齐版）
> **更新日期**: 2026-04-21
> **更新类型**: 闭环补齐 — 从"管好报告体系"到"数据采集→决策交付的完整闭环"

---

## 一、v1.0 回顾与盲区

v1.0 将方案从"教你怎么写报告"升级为"教你怎么管好整个报告体系"，新增了质量保障、运营管理、合规与防护三大模块，共11个Tab，覆盖了设计层和治理层。

```
v0.1  →  确立架构（四层报告架构）
v0.2  →  填充内容（六大能力模块）
v0.3  →  修复质量（Bug修复 + 内容补全）
v1.0  →  补齐治理（质量保障 + 运营管理 + 合规防护）
```

### 仍存在的盲区

| 盲区类别 | 具体问题 | 影响 |
|----------|----------|------|
| 输入层缺失 | 指标的数据从哪来、怎么采、怎么验 | 垃圾进垃圾出，下游分析和报告的可靠性无法保证 |
| 需求层缺失 | 写报告前没有标准化的需求沟通机制 | 70%的返工源于需求不清晰 |
| 组织层缺失 | RACI只管分工不管能力匹配 | 初级分析师写不了高管月报，高级分析师被日报占满时间 |
| 交付层缺失 | 报告写完了怎么送出去、谁看什么版本 | 报告写得再好送不到决策者手上，业务价值为零 |

## 三、逐项变更详情

### 变更 1/4：📡 新增「数据采集」Tab（独立模块）

**模块标识**: `s-collection`
**插入位置**: 导航栏第2位（体系总览之后、可视化仪表盘之前）
**变更类型**: 新增 Tab

#### 为什么新增

报告体系的质量上限由数据采集质量决定。v1.0 讲了指标口径、计算公式、质量检查，但没有讲这些指标的数据从哪来。如果埋点设计有问题，后面的所有分析都是垃圾进垃圾出。数据采集不属于现有任何模块——不合适塞进模板或KPI，也不属于质量保障（那是下游检查），因此需要作为独立模块存在。

#### 包含内容

| 子板块 | 说明 |
|--------|------|
| **事件命名规范** | 统一的 `模块_对象_动作` 格式（如 `order_create_success`、`user_login_fail`），含命名检查清单 |
| **埋点设计原则** | 6条核心原则：可追溯、唯一性、时间戳标准化、属性结构化、版本管理、去重机制 |
| **埋点质量监控** | 上线前必检项（覆盖率、格式校验、字段完整性）+ 上线后监控指标（日级埋点日志量波动、异常值检测） |
| **数据源接入标准** | 各类数据源（埋点日志、业务数据库、第三方API）的接入规范、格式要求、延迟预期 |

#### 与现有模块的关联

- **→ KPI体系**：口径注册表中的「数据来源」字段需要引用本模块的埋点规范
- **→ 质量保障**：质量自检中的「数据准确性」依赖本模块的埋点质量监控机制
- **→ 工具包**：SQL/Python模板的数据源引用需要遵循本模块的接入标准

#### HTML 实现说明

- 导航栏：在 `体系总览` 和 `可视化仪表盘` 之间插入 `<div class="nav-tab" onclick="sw('collection',this)">📡 数据采集</div>`
- 内容区：新增 `<div id="s-collection" class="section">`，包含4个 `.card`
- 每个card的结构与现有模块保持一致（`.ct` 标题 + `.cs` 副标题 + 内容主体）
- 不修改任何现有HTML结构，纯追加

### 变更 2/4：🧱 报告模板模块内追加「需求沟通指南」

**所属模块**: `s-templates`（报告模板）
**插入位置**: 该模块第一个 `.card` 之前
**变更类型**: 模块内追加子板块

#### 为什么追加

报告模板解决的是「写出来是什么样」，但实际工作中70%的报告问题是"业务方要的东西模糊或变了三次"，却没有一个标准化的需求沟通流程来对齐。需求没搞清楚就动手写，返工率极高。现有运营管理模块只写了"业务方提出需求→数据团队评估可行性"一句话，没有给出具体的沟通工具和模板。

#### 包含内容

| 子板块 | 说明 |
|--------|------|
| **需求澄清问卷模板** | 6个必问题：谁看、做什么决策、看多频繁、已有数据源、预期格式、截止时间 |
| **模糊需求翻译对照表** | 把业务方口头需求翻译成可执行的报告需求（如"看看最近数据怎么样" → 具体到哪些指标、什么时间范围、对比什么基准） |
| **需求评审标准** | 什么需求接、什么需求不接、什么需求需要裁剪（如"能不能把所有数据都放上去" → 退回去重做需求） |
| **报告范围蔓延防护** | 业务方不断增加指标/维度的应对策略 |

#### 与现有内容的关系

报告模板模块现有顺序：`6种报告结构模板` → `填写示例` → `版本管理规范`。需求沟通是「写报告之前」的环节，逻辑上属于模板模块的上游。追加在最前面作为"写报告前的准备"子板块，不改动任何已有内容的DOM顺序。

#### HTML 实现说明

在 `s-templates` section 内部，现有第一个 `.card` 之前插入新的 `.card` 元素。现有内容（6种模板卡片、填写示例、版本管理）原封不动。

### 变更 3/4：🔄 运营管理模块内追加「团队能力模型与分工」

**所属模块**: `s-ops`（运营管理）
**插入位置**: RACI矩阵card之后、异常响应SOPcard之前
**变更类型**: 模块内追加子板块

#### 为什么追加

RACI矩阵解决了「谁做什么」的问题，但没有解决「谁能做什么」的问题。实际中团队水平参差不齐，初级分析师写不了高管月报，高级分析师不应该被日报占满时间。需要把人放在合适的位置上，并给出成长路径。

#### 包含内容

| 子板块 | 说明 |
|--------|------|
| **分析师能力分层** | 初级（能执行）/ 中级（能分析）/ 高级（能洞察）三级定义，每级对应的核心能力、可独立负责的报告类型、典型工作产出 |
| **报告任务分配矩阵** | 什么级别的分析师负责什么类型的报告（如L0全自动日报→初级，L2人工主导高管月报→高级，L3定制专项→高级+负责人） |
| **技能矩阵与成长路径** | SQL/Python/统计/业务理解/沟通表达5维度的能力评估框架，从初级到高级的成长路径和培训资源 |
| **团队效率指标** | 人均报告产出、自动化覆盖率、返工率等团队层面的运营效率衡量 |

#### 与现有内容的关系

运营管理模块现有顺序：`报告生命周期` → `自动化分级` → `RACI矩阵` → `异常响应SOP`。能力模型和RACI是配套关系，插入到RACI之后、异常SOP之前。

插入后顺序变为：

```
报告生命周期 → 自动化分级 → RACI矩阵 → 团队能力模型（新） → 异常响应SOP → 报告分发与交付（新）
```

逻辑链路：有制度 → 有分工 → 有人才 → 有异常处理 → 有交付。

#### HTML 实现说明

在 `s-ops` section 内部，找到RACI矩阵对应的 `.card` 结束标签之后、异常SOP对应的 `.card` 开始标签之前，插入新的 `.card` 元素。

### 变更 4/4：🔄 运营管理模块内追加「报告分发与交付机制」

**所属模块**: `s-ops`（运营管理）
**插入位置**: 异常响应SOPcard之后（该模块最后一个card）
**变更类型**: 模块内追加子板块

#### 为什么追加

报告写得再好，送不到决策者手上或送到了但没人看，业务价值就是零。v1.0 覆盖了"怎么写"和"怎么管"，但完全没有讲"怎么送"。这是报告体系的最后一公里。

#### 包含内容

| 子板块 | 说明 |
|--------|------|
| **推送渠道选择矩阵** | 不同报告类型匹配不同推送渠道（邮件/飞书/企微/BI内嵌/自动化机器人），含时效性、互动性、权限控制对比 |
| **推送格式规范** | 不同渠道的报告格式要求（邮件摘要模板、飞书卡片格式、BI链接规范） |
| **权限与版本控制** | 同一份报告不同角色看到不同版本（如CEO看1页摘要版、总监看完整版、分析师看底层数据版），分发时的权限隔离实现方式 |
| **推送自动化配置** | 定时任务配置规范（日报每天9:00、周报每周一14:00），失败重试机制，推送失败通知 |
| **反馈收集机制** | 推送后嵌入反馈入口，自动收集读者对报告的改进建议 |

#### 与现有内容的关系

作为运营管理模块的最后一个card，代表日常运营流程的终点——报告写完、质量检查通过、异常处理完毕后，最终送达决策者。同时与「落地路线」模块中的效果度量形成呼应：分发机制提供了衡量"报告是否被有效使用"的触点。

#### HTML 实现说明

在 `s-ops` section 内部，找到异常响应SOP对应的 `.card` 结束标签之后，插入新的 `.card` 元素作为该模块的最后一个card。

## 五、变更理由总结

```
                    v1.0 覆盖范围
  ┌──────────────────────────────────────┐
  │  设计层：写什么、怎么写              │
  │  治理层：质量、合规、运营            │
  └──────────────────────────────────────┘
                      ↓ v1.1 补齐
  ┌──────────────────────────────────────┐
  │  输入层：数据从哪来（数据采集）  ← 新增  │
  │  需求层：写之前怎么沟通（需求指南）← 新增  │
  │  设计层：写什么、怎么写（已有）     │
  │  组织层：谁能做什么（能力模型）  ← 新增  │
  │  治理层：质量、合规、运营（已有）   │
  │  交付层：怎么送出去（分发机制）  ← 新增  │
  └──────────────────────────────────────┘
```

| 缺失环节 | 痛点 | 补齐方式 |
|----------|------|----------|
| 数据采集 | 指标口径依赖数据，但数据质量无人管 | 新增独立Tab |
| 需求沟通 | 70%返工源于需求不清晰 | 模板模块内追加 |
| 团队能力 | RACI只管分工不管能力匹配 | 运营模块内追加 |
| 报告分发 | 写完送不到，业务价值为零 | 运营模块内追加 |
